# App update functionality — Sparkle integration (design)

- **Date:** 2026-06-28
- **Status:** Approved (design); ready for implementation plan
- **Topic:** In-app auto-update for AppleNugs via the Sparkle framework

## Goal

Give AppleNugs a real update path: the app checks for new releases, tells the
user when one exists, and installs it in place with one click — without the user
hunting GitHub for a DMG. The mechanism must work for a **sandboxed,
hardened-runtime, notarized Developer ID** app distributed via GitHub Releases.

## Decision and why

Use **Sparkle 2.x** for full one-click auto-update. This was chosen over two
alternatives after an adversarial three-lens evaluation (ecosystem norms,
sandbox feasibility, total cost of ownership):

- **A hand-rolled in-app installer was rejected.** Under App Sandbox the app
  process *cannot* replace its own bundle in `/Applications`: no write access
  there, no ability to register a privileged helper, no atomic self-swap of a
  running `.app`. Apple's only sanctioned escape is a privileged installer XPC
  service launched out of the sandbox — which is precisely what Sparkle ships.
  Rolling that yourself reimplements Sparkle's hardest part (quarantine
  stripping, signature-continuity + EdDSA verification, privileged atomic
  replace, relaunch), each a security-sensitive footgun, while discarding its
  battle-tested code. Strictly more work and more risk.
- **A notify-only checker was considered and consciously declined.** A ~40–80
  line "new version available → open the download" nudge is the only *genuine*
  simplification (it drops silent install entirely, needs no entitlements/keys/
  appcast, and is equally safe because the user still installs the notarized,
  Gatekeeper-verified build). It was declined because the one-click,
  install-itself experience is wanted here. Recorded so the trade-off is
  explicit: we are paying Sparkle's recurring tax (signed appcast per release,
  one EdDSA key to guard) to buy hands-off updates.

Sparkle is the de-facto standard for this exact app profile and its setup cost
is **one-time configuration**, not recurring friction.

## Architecture

```
AppleNugsApp (@main)
  └─ owns SPUStandardUpdaterController (startingUpdater: true)
        ├─ Sparkle framework (SPM dependency)
        │     └─ Installer XPC service (runs OUTSIDE the sandbox; does the
        │        privileged /Applications replace + relaunch)
        └─ UpdaterViewModel (ObservableObject) mirrors `canCheckForUpdates`
              └─ "Check for Updates…" menu item (CommandGroup after .appInfo)

Feed:  app → SUFeedURL (raw.githubusercontent appcast.xml) → GitHub Release asset (DMG)
Trust: EdDSA(Ed25519) signature in appcast verified against SUPublicEDKey in the app
```

Sparkle's standard user driver provides the entire UI (update-available sheet,
download progress, install + relaunch). We add no custom update UI.

## Components / files

Three small, isolated additions. All other behavior is Sparkle's.

1. **`project.yml`** (source of truth; `.xcodeproj`/Info.plist/entitlements are
   regenerated by `xcodegen generate` and stay git-ignored — never hand-edit the
   outputs):
   - Add a `packages:` entry for Sparkle (SPM, pinned to a 2.x version).
   - Add `Sparkle` to the AppleNugs target `dependencies`.
   - Add the Info.plist keys and the entitlement (see below) as `properties`.

2. **`AppleNugs/App/AppleNugsApp.swift`** (extend the existing file):
   - In `init()`, construct the updater controller alongside the existing
     `URLCache` setup, and hold a `@StateObject` `updaterModel`
     (`UpdaterViewModel`, see file 3) seeded from the controller's updater.
   - Add `CommandGroup(after: .appInfo)` containing a "Check for Updates…"
     button, disabled while `updaterModel.canCheckForUpdates` is false.

   Representative shape (final code lives in the implementation plan):

   ```swift
   private let updaterController =
       SPUStandardUpdaterController(startingUpdater: true,
                                    updaterDelegate: nil,
                                    userDriverDelegate: nil)
   // …
   CommandGroup(after: .appInfo) {
       Button("Check for Updates…") { updaterController.checkForUpdates(nil) }
           .disabled(!updaterModel.canCheckForUpdates)
   }
   ```

3. **`AppleNugs/Updates/CheckForUpdates.swift`** (new; auto-included because the
   target sources the whole `AppleNugs/` tree):
   - `UpdaterViewModel: ObservableObject` with `@Published var canCheckForUpdates`,
     bound from `updater.publisher(for: \.canCheckForUpdates)`.
   - Main-actor-confined; Sparkle types are used only on the main actor so the
     file stays clean under `SWIFT_STRICT_CONCURRENCY = complete`.

## Info.plist keys (set via `project.yml` properties)

| Key | Value | Note |
| --- | --- | --- |
| `SUFeedURL` | `https://raw.githubusercontent.com/tsvb/applenugs/main/appcast.xml` | The feed Sparkle polls. Resolves only after the repo is public. |
| `SUPublicEDKey` | *(the Ed25519 public key)* | Safe to commit. |
| `SUEnableInstallerLauncherService` | `YES` | **Mandatory** for sandboxed apps. |

Deliberately **not** set:
- `SUEnableDownloaderService` — skipped because `com.apple.security.network.client`
  is already present (Sparkle's docs say to omit it in that case).
- `SUEnableAutomaticChecks` — left unset **on purpose**. Leaving it unset is what
  makes Sparkle show its first-launch "check for updates automatically?" prompt,
  which is the chosen behavior. (Setting it would suppress the prompt.)

## Entitlements (added to the existing sandbox block in `project.yml`)

Keep `com.apple.security.app-sandbox` and `com.apple.security.network.client`,
and add:

```
com.apple.security.temporary-exception.mach-lookup.global-name:
  - $(PRODUCT_BUNDLE_IDENTIFIER)-spks
  - $(PRODUCT_BUNDLE_IDENTIFIER)-spki
```

These let the sandboxed app reach Sparkle's installer XPC service. They are
`temporary-exception` entitlements: flagged in **App Store** review, but App
Store is already a non-goal (5.2.2) and they are fully fine for **Developer ID**,
the shipping path.

## Signing keys (one-time)

Run Sparkle's `generate_keys` once: it stores the **private** Ed25519 key in the
login Keychain and prints the **public** key. The public key goes into
`SUPublicEDKey`; the private key never enters the repo.

**Back the private key up outside the login Keychain** (export and store
securely). Losing it means no future update can be signed and existing installs
can never auto-update again — a real single-point-of-failure for a solo
maintainer. `generate_appcast` must run on the machine that holds the key, so
releases are tied to that machine's Keychain until the key is exported.

## Release workflow (extends the existing notarize pipeline)

The current per-release flow (bump version → archive → export → notarize →
staple → package → GitHub Release) gains two steps:

1. After notarizing + stapling, run Sparkle's **`generate_appcast`** over the
   build output. It signs the enclosure with the EdDSA private key and writes/
   updates `appcast.xml` (version, length, EdDSA signature, `minimumSystemVersion`
   = 14.0), using `--download-url-prefix`
   `https://github.com/tsvb/applenugs/releases/download/<tag>/` so the appcast
   points at the Release asset.
2. Commit + push `appcast.xml` (the raw URL of this file **is** the feed) and
   attach the enclosure to the GitHub Release.

**Enclosure = the DMG.** Sparkle supports DMG enclosures (mounts, copies the
`.app`), so the existing human-facing DMG download doubles as the update
artifact — one artifact, not two. `DISTRIBUTION.md` currently documents a
`.zip` (`ditto`) as the release artifact while the DMG was an ad-hoc step; this
work formalizes the DMG step in `DISTRIBUTION.md` so the documented flow matches
what Sparkle consumes.

**Versioning:** every release must **monotonically increment
`CURRENT_PROJECT_VERSION`** (→ `CFBundleVersion`). Sparkle compares *build
numbers*, not the marketing version string, and `generate_appcast` has
historically defaulted a missing/static build number to `1`, silently breaking
update detection. This is the single most common Sparkle footgun and is called
out in the runbook.

## Feed hosting and the dormant-until-public reality

The feed (`raw.githubusercontent.com`) and the Release-asset downloads only
resolve once the repo is **public** (private repos require a token Sparkle won't
send). Until then, "Check for Updates" fails gracefully (Sparkle reports it
could not reach the feed; no crash). The plumbing activates the moment the repo
is publicized — no code change needed.

Two operational gotchas of the raw-URL feed, documented in the runbook:
- GitHub's CDN caches the raw file ~5 minutes, so a just-pushed update is not
  instantly visible.
- Go-live is coupled to remembering to push `appcast.xml` *after* the Release
  asset is uploaded; an appcast that references a not-yet-uploaded asset points
  at a 404. Release discipline: upload the asset first, then push the appcast.

## Verified risks and mitigations

| Risk | Mitigation |
| --- | --- |
| Forgot to bump `CFBundleVersion` → no update detected | Runbook step + pre-ship check that build number increased. |
| Lost EdDSA private key → can never sign updates again | Export + back up the key outside the Keychain at setup time. |
| appcast pushed before asset uploaded → 404 feed | Documented ordering: asset first, then appcast push. |
| Repo still private → feed 404 | Expected; checker fails gracefully; activates on publicize. |
| Strict-concurrency warnings from Sparkle usage | Keep updater controller + view model main-actor-confined. |

Note: EdDSA is **not** the only thing protecting the user — every build is
Developer-ID-signed, notarized, stapled, and Gatekeeper-verified on open. EdDSA
is Sparkle's requirement for its silent-install path, an additive trust layer.

## Testing / verification

Sparkle's flow is integration-level, not unit-testable in isolation. Acceptance:

1. **Clean build** with the SPM dependency + new entitlements under
   `SWIFT_STRICT_CONCURRENCY = complete` (Xcode 26.5, fresh `-derivedDataPath`)
   is warning-free.
2. **Menu item** "Check for Updates…" appears under the app menu and is enabled
   once the updater is ready.
3. **End-to-end update check** against a temporary public feed (a throwaway
   gist or a `file://` appcast) with a build-N and build-N+1: confirm Sparkle
   detects, downloads, verifies the EdDSA signature, installs, and relaunches.
   This is done before the real feed goes live so the private-repo state does
   not block verification.

## Scope / non-goals (YAGNI)

**In:** signed full-DMG updates, automatic checks with first-launch prompt,
manual "Check for Updates…", Sparkle's standard UI, release tooling + docs.

**Out (deferrable):** delta/incremental updates, a beta/pre-release channel,
custom in-app release-notes styling (GitHub release notes suffice), and
CI-automated signing (releases stay a local signed step on the maintainer's
machine, matching today's flow).
